## 第一章：产品概述

**接团通** 是一款 **面向中小地接社的接团与结算管理 SaaS**，让 **计调、老板和财务** 能够 **以团号为中心管住每个团的收支、结算节点和合作方往来**，而无需 **在 Excel 台账、微信群聊天记录和纸质记账本之间来回核对**。

### 1.1 差异化对比表

最相关的竞品有三类：① 路书星球类行程工具；② Excel + 微信（绝大多数中小地接社的现状）；③ 传统旅行社 ERP（本地部署、按年收费）。

| 功能 | 竞品 | 本产品 | 实现方式 |
|------|------|--------|----------|
| 以团号为中心的台账 | 路书以「行程/线路」为中心，行程编辑是主线，钱是附属 | 接团单（团号）是唯一中心，行程只是一个文本字段 | `projects` 表，`group_no` 在租户内唯一索引，所有收支/节点外键挂团 |
| 收付款节点与逾期感知 | Excel 靠手动标黄标红；ERP 有应收应付但藏在财务模块深处 | 每团可挂多个应收/应付节点，工作台首屏直接列出「逾期 N 笔 / 7 日内到期 M 笔」 | `payment_schedules` 表 + 工作台聚合查询，overdue 由 `due_date < today AND status='pending'` 实时推导，不存冗余状态 |
| 每团毛利实时可见 | ERP 月底跑报表；Excel 要手动拉公式 | 录一笔收支，团详情和列表的毛利立即更新 | `transactions` 金额一律整数分存储，SQL `SUM` 聚合，无浮点误差 |
| 零部署、零年费起步 | 传统 ERP 本地部署 + 数千到数万年费；路书按席位收费 | 多租户 SaaS，跑在 Cloudflare 免费额度内 | Pages + D1 + KV，单 Worker 同时服务前端静态资源与 API |
| 组团社/供应商关系沉淀 | 微信通讯录 + 个人记忆，人走关系断 | 合作方/供应商档案，自动关联历史团与累计往来金额 | `partners` / `suppliers` 表，详情页聚合该实体名下全部接团单 |

注：行程编辑、地图、获客分销是路书的强项，本产品**明确不做**（见 1.3），不在此表凑数。

### 1.2 三类用户画像

**① 地接社老板（王总，45 岁，公司 8 人）**
- 核心目标：随时知道「这个月接了多少团、赚了多少、谁还欠我钱」。
- 最大不满：现状是问计调要 Excel，版本永远不是最新的；月底对账要花两天。
- 切换理由：打开工作台 3 秒看到本月团数、毛利、逾期应收——这一个页面就够了。

**② 计调（小李，27 岁，同时跟 10+ 个团）**
- 核心目标：接到组团社的团，快速建单、排供应商、记成本，不漏任何一笔。
- 最大不满：Excel 多人同时编辑会互相覆盖；微信里翻三个月前的报价要翻半小时。
- 切换理由：新建接团单 30 秒完成，组团社和供应商从档案里直接选，不用重复打字。

**③ 财务/会计（陈姐，38 岁，兼职给三家小地接社做账）**
- 核心目标：每个团的收支清楚、有据可查，月底能直接出对账数据。
- 最大不满：计调记的流水缺日期、缺对方名称，金额还有手滑多个零的。
- 切换理由：每笔收支强制关联团号和分类，金额输入框只收两位小数，导出就是干净的对账单（v1.5）。

### 1.3 可行性边界

| 在范围内 | 原因 |
|----------|------|
| 多租户注册/登录、员工与邀请码 | SaaS 模式的地基；D1 + KV 足以支撑 |
| 接团单（团号、状态流、人数、日期、备注） | 产品核心实体 |
| 合作方（组团社）与供应商档案 | 地接社的两端关系，结构简单、价值高 |
| 每团收支明细与毛利核算 | 整数分 + SUM 聚合，技术上零风险 |
| 收付款节点与逾期提醒（站内） | 纯查询推导，无需定时任务也能实现 |
| Excel/CSV 导出（v1.5） | 前端生成，不占服务端资源 |

| 明确排除 | 原因 |
|----------|------|
| 行程编辑器、地图、POI 库 | 路书的核心壁垒，重资产功能，与「管钱管团」主线无关 |
| 附件/合同上传 | 需引入 R2，v1 控制复杂度；先用备注字段过渡 |
| 微信/短信通知 | 需企业资质与模板审核，免费额度外；v1 用站内提醒 |
| OTA / 机票酒店库存对接 | 中小地接社以线下协议价为主，对接成本远大于收益 |
| 发票与税务 | 合规复杂度高，用户现阶段用线下流程 |
| 离线可用 | D1 为单一数据源，离线同步冲突处理成本不可控 |
| 散客 CRM（C 端客户管理） | 已确认目标用户是地接社，「客户」即组团社，不是散客 |

### 布局逻辑说明

地接社的工作节奏是「以团为单位的多线程」：计调上午建团、下午记账、随时被老板问数。侧栏固定 7 个入口，全部一级可达，没有二级菜单——因为目标用户（中小社员工）对「藏在菜单里的功能」的容忍度接近零。顶栏全局搜索直达团号是高频动作（微信里组团社甩来一句「JTT-0612 那个团怎么样了」，计调要 3 秒内打开）。内容区限宽 1200px 是因为核心内容是表格和表单，过宽反而降低可读性。

**b) 交互流程**

```
用户提交注册表单
    |
    v
前端 zod 校验（手机号格式 / 密码 ≥ 8 位）
    |
    +-- 校验失败 --> 行内红字提示，不发请求
    |
    v
POST /api/auth/register
    |
    +-- 手机号已存在 --> 409，提示「该手机号已注册，去登录」
    |
    +-- 成功 --> 创建 organization + owner 用户（同一 D1 batch）
                 签发 access JWT（15min）+ refresh token 写 KV（7 天）
                 跳转 /dashboard

登录失败路径：
连续 5 次密码错误 --> KV 记录计数 --> 锁定 15 分钟，
                     提示「尝试次数过多，请 15 分钟后再试」
邀请码失败路径：
邀请码已用尽/过期 --> /join 页直接显示「邀请已失效，请联系管理员重新生成」
```

**c) 状态清单**

| 状态 | 触发条件 | 视觉标识 | 退出条件 |
|------|----------|----------|----------|
| 未登录 | 无有效 token | 重定向 /login | 登录成功 |
| 已登录 | access JWT 有效 | 正常应用壳 | 退出 / refresh 失效 |
| 静默续期中 | access 过期但 refresh 有效 | 无感（请求自动重试一次） | 续期成功或失败 |
| 登录锁定 | 5 次密码错误 | 登录按钮置灰 + 倒计时文案 | 15 分钟后 |
| 邀请失效 | 码过期/用尽 | /join 页错误卡片 | 获得新邀请链接 |

**d) 依赖关系**

- 写入：`organizations`、`users`（D1）；refresh token、限流计数（KV）。
- 读取：`invites`（加入企业时校验）。
- 下游：所有模块依赖本模块产出的 `org_id` + `user_id` 上下文做租户隔离。

**e) 待决问题**

1. 手机号是否需要短信验证码实名？v1 决定**不做**（免费额度无短信通道），风险是可被批量注册——靠邀请制内推 + 后续观察。此决策若上线后出现垃圾注册，需补人机验证（Turnstile，Cloudflare 免费）。

## 第四章：超越竞品的差异化功能

### 4.1 收付款节点驱动的「结算感知」工作台

**竞品为何没有**：路书类工具的商业模式建立在「行程内容生产」上，财务只是行程的附属字段，其架构里没有「应收应付的时间维度」这一实体；传统 ERP 有应收应付，但其设计是给专职财务月底批处理用的，逾期信息埋在报表里，而中小地接社根本没有专职财务每天去翻报表。这是商业模式和目标组织规模差异导致的结构性盲区。

**本产品如何实现**：用独立的 `payment_schedules` 表（团 ID、方向、金额分、到期日、状态），工作台聚合查询 `status='pending'` 按 `due_date` 排序，逾期由日期比较实时推导而非定时任务写状态。选这个方案而不是 Cron Trigger 批量标记，因为推导式零运维、零延迟、且免费额度内不占 Cron 配额；代价是每次查询多一个日期比较，在千行量级可忽略。

**交互流程**：

```
计调建团时顺手添加节点「定金30% 到期06.10」「尾款 到期06.20」
    |
    v
06.11 老板打开工作台
    |
    v
首屏卡片「逾期应收 ¥23,500 / 3 笔」红色高亮
    |
    v
点击 --> 节点列表（按逾期天数降序）--> 点行进入团详情
    |
    v
打电话催款 --> 到账 --> 点「标记已收」--> 自动生成收入明细
    |
    v
工作台数字即时更新，逾期卡片消除红色
```

**风险与应对**：风险是用户嫌「建节点」多一步操作，节点数据稀疏导致功能空转。应对：新建团表单内提供「定金+尾款」一键模板（按比例自动算金额）；若上线后节点创建率仍低于 30%，备选方案是从收支明细反推（录入定金时询问「尾款何时收？」）。

### 4.2 团号为中心的统一信息模型

**竞品为何没有**：Excel 的本质是表格而非数据库，团、钱、合作方之间没有外键，只能靠人肉对齐行；路书的中心实体是「行程」，同一行程可复用于多个团，导致它的数据模型天然不适合做单团核算。两者都不是「没想到」，而是其基础数据模型决定了改造成本极高。

**本产品如何实现**：`group_no` 在租户内全局唯一（D1 复合唯一索引 `(org_id, group_no)`），收支、节点、合作方往来全部通过 `project_id` 外键挂团，顶栏搜索直达。团号自动生成 `JTT-YYYYMMDD-NN` 但允许手改，因为很多地接社已有自己的编号习惯，强制格式会成为迁移阻力。

**交互流程**：

```
微信里组团社问「JTT-20260601-02 尾款发你了」
    |
    v
计调在顶栏输入 0601 --> 下拉即时匹配（防抖 300ms）
    |
    v
回车进入团详情 --> 收付款 Tab --> 标记尾款已收
    |
    v
全程 < 10 秒，无需翻 Excel 文件名找「6月版最终版(2).xlsx」
```

**风险与应对**：风险是手改团号导致重复冲突的体验问题。应对：前端失焦即查重（而非提交时才报错）；若用户普遍手改，说明自动格式不符合行业习惯，备选方案是开放租户级前缀配置（已在 3.5 待决问题中预留）。

### 4.3 供应商成本溯源

**竞品为何没有**：ERP 的供应商模块面向采购合同管理，录入字段多达数十个，中小地接社用不动；Excel 里供应商只是支出行里的一个文本格子，同一家酒店被写成「悦榕」「西街悦榕」「悦榕酒店」三种名字，永远无法聚合。这是「重工具过重、轻工具无结构」的中间地带空缺。

**本产品如何实现**：供应商档案只有 4 个字段（名称、类型、联系人、电话），支出明细通过下拉**选择**而非手输供应商，从源头消灭别名问题。供应商详情页用一条 `SUM` 聚合出累计成本与未付金额。刻意不做采购合同、协议价库——那是 ERP 的领域。

**交互流程**：

```
记支出 --> 供应商下拉 --> 输入「悦」--> 命中「阳朔西街悦榕酒店」
    |                          |
    |                          +-- 无命中 --> 「+ 新建供应商"悦"」内联创建
    v
保存 --> 年底打开供应商详情
    |
    v
「本年度在悦榕消费 ¥128,400 / 23 个团」--> 拿着数据去谈明年协议价
```

**风险与应对**：风险是用户图省事不选供应商（该字段选填）。应对：选填是刻意的（强制会逼用户乱选污染数据）；若关联率低，在支出分类为「酒店/门票/用车」时将下拉前置为表单第二个字段，用顺手程度而非强制来提升关联率。

```jsonc
  "date": "2026-06-12",        // 必填，发生日期，统计归属依据
  "note": "大床*12 标间*4",     // 选填，≤ 200 字符
  "created_by": "usr_a1b2",    // 必填
  "created_at": 1749700000
}

// payment_schedules — 收付款节点
{
  "version": 1,
  "id": "sch_k1l2",
  "org_id": "org_8f2k",        // 必填
  "project_id": "prj_g7h8",    // 必填
  "direction": "receivable",   // 必填，receivable(应收)|payable(应付)
  "title": "尾款",              // 必填，1-30 字符
  "amount_cents": 3402000,     // 必填，正整数
  "due_date": "2026-06-20",    // 必填；逾期 = due_date < today 且 status=pending，实时推导不落库
  "status": "pending",         // 必填，pending|done，默认值: pending
  "done_date": null,           // status=done 时必填，实收/实付日期
  "done_txn_id": "txn_…",      // 选填，联动生成的明细 ID，便于撤销时级联提示
  "created_at": 1749700000
}
```

**设计决策说明**：① 不设独立的「账户/钱包」实体——地接社关心的是「每个团赚不赚、谁欠谁」，不是银行账户余额，加账户体系会把产品推向记账软件红海。② `transactions.org_id` 冗余存储是刻意的：全局流水页直查一张表，避免 JOIN projects，D1 单表索引扫描性能可预期。③ 逾期不落库、纯推导，消灭「定时任务漏跑导致状态错」这一整类 bug。④ 刻意排除：审计日志表（v1 用 `created_by` 字段达到 80% 追溯效果）、软删除（`status=disabled` 已覆盖需求场景）。

## 第六章：技术架构

```
+--------------------------------------------------------------------------+
| 浏览器层：React SPA                                                       |
| 职责：路由、表单校验(zod)、乐观更新、金额分<->元换算只在展示层发生           |
+--------------------------------------------------------------------------+
                          | HTTPS / JSON，Authorization: Bearer <JWT>
                          v
+--------------------------------------------------------------------------+
| 边缘层：Cloudflare Pages Functions（Hono 单 Worker）                      |
| 职责：JWT 验签、org_id 注入与强制过滤、zod 二次校验、限流、业务规则          |
| 约束：任何 SQL 必须带 org_id 条件——由 DAO 层统一拼接，handler 不写裸 SQL    |
+------------------------------+-------------------------------------------+
                |                              |
                v                              v
+------------------------------+  +---------------------------------------+
| D1 (SQLite)                  |  | KV                                    |
| 职责：唯一事实源              |  | 职责：refresh token(7d TTL)、          |
| 8 张表 + 复合索引             |  | 登录限流计数(15m)、dashboard 缓存(60s) |
| 迁移：drizzle-kit 生成 SQL    |  | 约束：KV 数据丢失不影响正确性，只影响    |
|                              |  | 体验（缓存重建 / 重新登录）              |
+------------------------------+  +---------------------------------------+
```

| 库名 | 用途 | 为何优于替代方案 | 大致包体积 |
|------|------|-----------------|-----------|
| hono | 边缘路由框架 | 为 Workers 设计，冷启动开销远小于 Express 类；中间件模型适合做 org_id 强制注入 | ~14KB |
| drizzle-orm | D1 ORM + 迁移 | 原生支持 D1 驱动；类型安全的查询构建器使「漏带 org_id」在编译期可被封装拦截；Prisma 不支持 D1 免费层友好部署 | ~30KB（按需） |
| zod | 前后端共享校验 | 同一 schema 两端复用，金额/日期规则只写一次；比 yup 的 TS 推导更完整 | ~14KB |
| react + react-router | SPA 框架 | 团队与 AI 生态最成熟；本产品页面数 ~12，不需要 SSR（数据全在登录墙后，无 SEO 诉求） | ~45KB + ~12KB |
| @tanstack/react-query | 服务端状态 | 内置缓存失效、乐观更新、请求去重，手写这些是 bug 高发区 | ~13KB |
| dayjs | 日期处理 | 到期日/逾期天数计算；比 date-fns 体积小且 API 对中文团队更直觉 | ~3KB |

不引入组件库（用 CSS 原生变量 + 少量手写组件），因为参考 UI（路书风格青绿主题）定制度高，antd 类库的覆盖样式成本反而更高。图表 v1 只有工作台一处趋势，用原生 SVG 手绘折线，不引入 echarts（~400KB）。

**最大架构风险**：D1 免费层的限制（单库 500MB、每日读写行数配额）与单区域写入延迟。应对：本产品数据天然轻量（一个 10 人地接社年数据量 < 5MB），按当前模型 500MB 可支撑数百租户；所有列表强制 cursor 分页（禁 OFFSET）+ 复合索引，把每请求读行数控制在两位数；若单库逼近上限，预案是按 org 分库（D1 支持多库绑定），数据模型已按 org_id 隔离，迁移只需搬运不需改 schema。

## 第九章：开发优先级

**P0 — 没有这个，产品根本无法使用。** 交付标准：功能可用，不需要完美。

- 注册（建企业）/ 登录 / JWT + refresh 续期
- 合作方 CRUD（建团的前置依赖）
- 接团单：新建、列表（基础筛选）、详情、状态流转
- 收支明细 CRUD + 团内三合计（应收/成本/毛利）

**P1 — 没有这个，用户第一次体验后不会回来。** 交付标准：功能完整，体验流畅。

- 工作台（4 统计卡 + 待办节点 + 最近团）
- 收付款节点 + 逾期推导 + 工作台联动 + 标记联动生成明细
- 供应商档案 + 支出关联
- 邀请码加入企业 + 员工列表/停用
- 全局搜索（团号/合作方）+ 列表 cursor 分页

**P2 — 有了这个，用户会把产品推荐给别人。** 交付标准：稳定且有辨识度。

- 合作方/供应商详情页的往来聚合（历史团 + 累计金额）
- XLSX/CSV 导出（8.1 全部三种）
- 定金+尾款一键节点模板；收支「复制为新支出」
- 工作台毛利趋势图（近 6 个月）

**P3 — 有了这个，一部分用户会付费或强烈倡导。** 交付标准：精致且有完整文档。

- 批量对账单导出（zip）
- 团号前缀自定义、member 细粒度权限
- 移动端布局完整适配（≤ 768px）
- 数据导入（Excel 历史台账迁移向导）

## 第十一章：开发者交接说明

本章写给将要实现本产品的你（AI 编程助手）。

### a) 实现顺序建议

按此顺序构建：**① schema + 迁移 → ② 认证与 org 中间件 → ③ 合作方 → ④ 接团单 → ⑤ 收支 → ⑥ 节点 → ⑦ 工作台 → ⑧ 供应商 → ⑨ 员工/邀请 → ⑩ 导出**。理由：org_id 注入中间件是所有 API 的地基，必须最先稳定；合作方是建团的必填外键，先于接团单；工作台是纯聚合消费方，等数据生产端（⑤⑥）就绪后做，否则只能对着空数据开发；员工/邀请独立性强，可随时插入间隙。每完成一个模块就让它端到端可用（API + 页面），不要先做完所有 API 再做前端。

### b) 最可能导致返工的三个决策

1. **金额换算的位置**。安全默认：API 进出全部用 `_cents` 整数，元<->分换算只发生在前端展示组件这一层。改向信号：如果你发现自己在后端写 `* 100` 或 `/ 100`，说明换算层泄漏了，立即收拢。
2. **org_id 过滤的实现方式**。安全默认：写一个 DAO 层（或 drizzle 查询包装器），所有查询必须经它注入 `org_id`，handler 禁止裸写 SQL/查询构建器。改向信号：任何一个接口测试中出现跨租户数据，停下重构这一层，而不是在 handler 里补 where。
3. **节点与明细的联动**（3.6 待决问题的已定方案）。安全默认：标记节点 done 时在同一 D1 batch 内创建明细并回填 `done_txn_id`。改向信号：如果撤销/删除的级联逻辑让你写出超过 ~50 行的状态机，简化为「联动创建，撤销时只提示不自动删」。

### c) 哪里要严格，哪里可以灵活

| 章节 | 性质 |
|------|------|
| 1.3 范围边界 | **约束**——不得实现排除项 |
| 第三章各模块的状态清单与失败路径 | **约束**——状态枚举与流转规则不得增删 |
| 第三章 ASCII 图的具体像素/布局 | 建议——以原型 `prototype/index.html` 风格为准，可优化 |
| 第五章数据模型字段与命名 | **约束**——字段语义与 `_cents` 规则不得偏离；可增加索引等实现层细节 |
| 第六章依赖库 | 建议——同能力可替换，但替换需满足体积与 D1 兼容前提 |
| 第七章快捷键/菜单/文案 | 建议——文案可润色，行为语义保持 |
| 第九章优先级分层 | **约束**——P0 未完不得开始 P2 |
| 第十章性能数字 | **约束**——劣化阈值是验收红线 |

### d) 已知的未知项

1. **团号前缀自定义的真实需求强度**未验证（3.5 待决问题）。v1 写死 `JTT-`，不要提前做配置化。
2. **D1 免费层每日行读配额在真实使用下的消耗速度**未实测。工作台聚合是读大户，上线后第一周必须看 Cloudflare 控制台用量；若超预期，把 dashboard 缓存 TTL 从 60s 提到 300s 是第一杠杆。
3. **「合作方既是组团社也是供应商」的双重身份**（例如同行互相甩团）本 PRD 未建模——v1 接受在两张表各建一条记录的轻微冗余，不要尝试设计统一的 contact 实体，那是过度抽象。
4. **多人同时编辑同一个团的基本信息**没有锁机制，最后写入胜出。v1 接受此行为（团队 ≤ 10 人，概率低），不要实现版本号/CRDT。
