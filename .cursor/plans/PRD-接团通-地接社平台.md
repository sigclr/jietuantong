## 第一章：产品概述

**接团通** 是一款 **面向中小地接社的接团与结算管理 SaaS**，让 **计调（OP）、老板和财务** 能够 **以团号为中心管住每个团的收支、结算节点和合作方往来**，而无需 **在 Excel 台账、微信群聊天记录和纸质记账本之间来回核对**。

产品演进逻辑（行业共识）：**计调流程标准化** → **后台系统一体化** → **导游/司机移动协同** → **管理层数据驱动决策**。v1 聚焦中间两步中的「接团 + 核算 + 结算」闭环；移动协同与 BI 深化放在后续版本（见 1.6）。

### 1.1 差异化对比表

最相关的竞品有三类：① 路书星球类行程工具；② Excel + 微信（绝大多数中小地接社的现状）；③ 传统旅行社 ERP（本地部署、按年收费）。

| 功能 | 竞品 | 本产品 | 实现方式 |
|------|------|--------|----------|
| 以团号为中心的台账 | 路书以「行程/线路」为中心，行程编辑是主线，钱是附属 | 接团单（团号）是唯一中心，行程只是一个文本字段 | `projects` 表，`group_no` 在租户内唯一索引，所有收支/节点外键挂团 |
| 收付款节点与逾期感知 | Excel 靠手动标黄标红；ERP 有应收应付但藏在财务模块深处 | 每团可挂多个应收/应付节点，工作台首屏直接列出「逾期 N 笔 / 7 日内到期 M 笔」 | `payment_schedules` 表 + 工作台聚合查询，overdue 由 `due_date < today AND status='pending'` 实时推导，不存冗余状态 |
| 每团毛利实时可见 | ERP 月底跑报表；Excel 要手动拉公式 | 录一笔收支，团详情和列表的毛利立即更新 | `transactions` 金额一律整数分存储，SQL `SUM` 聚合，无浮点误差 |
| 零部署、零年费起步 | 传统 ERP 本地部署 + 数千到数万年费；路书按席位收费 | 多租户 SaaS，跑在 Cloudflare 免费额度内 | Pages + D1 + KV，单 Worker 同时服务前端静态资源与 API |
| 组团社/供应商关系沉淀 | 微信通讯录 + 个人记忆，人走关系断 | 合作方/供应商档案，自动关联历史团与累计往来金额 | `partners` / `suppliers` 表，详情页聚合该实体名下全部接团单 |

注：行程编辑、地图、获客分销是路书的强项，本产品**明确不做**（见 1.3），不在此表凑数。

### 1.2 四类用户画像

**① 地接社老板（马总，45 岁，公司 8 人）**
- 核心目标：随时知道「这个月接了多少团、赚了多少、谁还欠我钱」。
- 最大不满：现状是问计调要 Excel，版本永远不是最新的；月底对账要花两天；看不到实时盈亏与现金流风险。
- 切换理由：打开工作台 3 秒看到本月团数、毛利、逾期应收——这一个页面就够了。

**② 计调 / OP（阿财，27 岁，同时跟 10+ 个团）**
- 角色定位：地接社的「中枢」——资源整合、产品交付、多方协调（见 1.5）。
- 核心目标：接到组团社的团，快速建单、排供应商、记成本，不漏任何一笔；减少电话/微信重复确认。
- 最大不满：Excel 多人同时编辑会互相覆盖；微信里翻三个月前的报价要翻半小时；行程变更后成本要手工重算。
- 切换理由：新建接团单 30 秒完成，组团社和供应商从档案里直接选，不用重复打字；团号搜索 3 秒定位。

**③ 财务/会计（王姐，38 岁，兼职给三家小地接社做账）**
- 核心目标：每个团的收支清楚、有据可查，月底能直接出对账数据；应收应付账龄一目了然。
- 最大不满：计调记的流水缺日期、缺对方名称，金额还有手滑多个零的；与供应商结算周期长、对账工作量大。
- 切换理由：每笔收支强制关联团号和分类，金额输入框只收两位小数，导出就是干净的对账单（v1.5）。

**④ 导游 / 司机（努尔兰，外部协作者，v2+ 移动端覆盖）**
- 核心目标：行程、名单、确认单一处查看；报账有凭证、不丢票。
- 最大不满：行程变更靠电话通知，信息滞后；费用靠纸质票据，回社后补录易遗漏。
- 切换理由（远期）：导游端小程序查看团信息、移动报账、在线审批——v1 不实现，但数据模型预留 `created_by` 与团维度归属，避免返工。

### 1.3 四类角色痛点对照（行业方案归纳）

| 角色 | 典型痛点 | 接团通 v1 如何缓解 | 后续版本补强 |
|------|----------|-------------------|--------------|
| 计调 OP | 电话微信轰炸；手工录单易错；响应客户慢 | 团号中心建档、档案快选、顶栏搜索、节点模板 | 询价报价一键生成、确认单派发、团内变更联动重算毛利 |
| 老板 | 无法实时看盈亏；结算周期长；内控难 | 工作台 4 卡 + 逾期应收首屏、每团毛利即时 | 计划 vs 实际毛利对比、BI 看板、异常团预警 |
| 财务 | 手工对账重；付款周期长；差错风险 | 收支挂团号、节点账龄、对账单导出 | 应收应付账龄报表、批量结算单、多级审批 |
| 导游/司机 | 行程变更滞后；手工报账；票据易丢 | —（v1 不含移动端） | 导游小程序：行程/名单/凭证、移动报账 |

### 1.4 计调（OP）工作阶段与系统映射

地接行业中 **OP（Operator，计调）** 是接团业务的核心执行者，系统 v1 必须优先服务其高频闭环，而非一次性覆盖全链路。

```
行前行前                    行中                        行后
+-- 询价/报价（v2）         +-- 行程/人数变更（v2）      +-- 单团盈亏结算（v1 ✓）
+-- 成本测算（v1 手工记账）  +-- 应急协调（线下）         +-- 供应商对账（v1 节点+导出）
+-- 资源预订（v1 供应商档案） +-- 导游协同（v2 小程序）    +-- 质量反馈（v3）
+-- 导游派遣（v2）           +-- 实时成本追加（v1 ✓记账）
+-- 建团/确认（v1 ✓）
```

**v1 设计原则（面向 OP）**：
1. **流程标准化**：团状态流、收支分类、节点方向（应收/应付）固定枚举，减少「每人一套 Excel 列名」。
2. **能自动就不手填**：团号自动生成、节点标记联动生成收支明细、定金+尾款模板（P2）。
3. **多端同步（渐进）**：v1 仅 Web；v2 起导游端只读 + 报账，变更仍以 OP 在 Web 端录入为权威源。

### 1.5 业务能力全景与版本路线图

参考地接社数字化管理方案，将业务能力分为四大模块，并明确 **v1 做深做透的范围** 与 **分期交付**：

| 业务模块 | 子能力 | v1（P0–P1） | v1.5–P2 | P3 及以后 |
|----------|--------|-------------|---------|-----------|
| **接团与 OP 管理** | 接团单建档、状态流 | ✓ | 团内变更日志 | 行中变更自动重算、质量反馈 |
| | 团款组成（分项单价录入） | ✓（v2.2 纳入，v2.3 加减项） | 报价单导出 | 与确认单联动 |
| | 拼出团（转交同行） | ✓（v2.2–v2.5，v2.6 录入合一） | 拼出毛利/差价核算 | 同行往来对账 |
| | 询价/报价、行程确认单 | — | 报价单 XLSX/PDF 导出 | 一键生成确认单并派发供应商 |
| | 导游派遣与协同 | — | — | 导游端小程序 + 一键派团 |
| **供应商管理** | 资源档案（酒店/用车/餐饮/门票/导服） | ✓ 简易名录 | 详情页成本聚合 | 协议价/季节价库、库存余量预警 |
| | 供应商评估 | — | — | 基于团后反馈的自动评分 |
| **合作方/渠道** | 组团社档案、账期、未结清 | ✓ | 往来汇总、对账单 | 授信额度、区域业绩统计 |
| | OTA / 大客户 CRM | — | — | 按付费意愿评估，非 v1 客群 |
| **收入与财务** | 单团收支、毛利 | ✓ | 计划毛利 vs 实际毛利 | 异常团自动标红 |
| | 收付款节点、逾期感知 | ✓ | 账龄分析视图 | 批量结算、自动催款提醒（需短信） |
| | 导游报账审批 | — | — | 移动提交 + 多级审批 |

**转型路径（与行业方案一致）**：

```
计调流程标准化（v1 团号+状态+记账）
        |
        v
后台一体化管理（v1 合作方/供应商/财务流水）
        |
        v
导游移动工具（v2 小程序）
        |
        v
老板数据驾驶舱（v2 趋势图 → v3 BI）
```

### 1.6 可行性边界

| 在范围内 | 原因 |
|----------|------|
| 多租户注册/登录、员工与邀请码 | SaaS 模式的地基；D1 + KV 足以支撑 |
| 接团单（团号、状态流、人数、日期、备注） | 产品核心实体；对应 OP「建团/确认」 |
| 团款组成（分项单价 + 加减项 + 备注，非总价） | 客户反馈：多行报价，每行可加可减（如优惠退费、增加住宿） |
| 拼出团（转交同行 + 明细分项 + 顶栏摘要 + 应付节点 + 分次结账） | 客户/Gemini：拼出价顶栏可见，可分次对同行结账 |
| 合作方（组团社）与供应商档案 | 地接社 B2B 两端关系；对应渠道与供应链基础数据 |
| 每团收支明细与毛利核算 | 整数分 + SUM 聚合；对应「单团盈亏」核心财务能力 |
| 收付款节点与逾期提醒（站内） | 纯查询推导；对应应收应付账龄的 v1 最小实现 |
| Excel/CSV 导出（v1.5） | 前端生成；对应财务对账与报价单导出的第一步 |

| 明确排除（v1 不做，且近期无计划） | 原因 |
|----------|------|
| 行程编辑器、地图、POI 库 | 路书的核心壁垒；v1 用备注字段承载行程文本即可 |
| 附件/合同上传 | 需引入 R2，v1 控制复杂度 |
| 微信/短信通知 | 需企业资质；v1 用站内提醒，逾期靠工作台 |
| OTA / 机票酒店库存对接 | 中小地接社以线下协议价为主，对接 ROI 低 |
| 发票与税务 | 合规复杂度高 |
| 离线可用 | D1 单一数据源，离线同步成本不可控 |
| 散客 CRM（C 端客户管理） | 目标客户是地接社，「客户」即组团社 |

| 后续版本规划（曾出现在行业方案中，非 v1 排除） | 预计版本 | 前置依赖 |
|----------|----------|----------|
| 询价/报价单一键生成（PDF/Word） | v1.5–P2 | 团内收支结构稳定、导出能力 |
| 计划毛利 vs 实际毛利、异常团标识 | P2 | 接团单「团款组成」加减汇总为计划收入（见 3.5.1、4.4） |
| 确认单派发（酒店/车队/景区） | P2–P3 | 供应商档案 + 通知通道 |
| 导游/司机小程序（行程、名单、移动报账） | P3 | 账号体系扩展、移动端 |
| 供应商协议价/季节价、库存预警 | P3 | 供应商模块深化 |
| 老板 BI 看板（收入/利润/现金流/风险） | P3 | 工作台趋势图 + 多维度聚合 |

### 1.7 客户反馈修订摘要（2026-06-16）

> 来源：[客户反馈截图](../../参考/客户反馈/PixPin_2026-06-16_13-56-48.png)、[澄清说明](../../参考/客户反馈/PixPin_2026-06-16_13-57-54.png)

首批试用客户（新疆地接社场景）对 **建团基本信息** 与 **同行拼出** 提出两项结构性调整，v2.2 起纳入 P1 范围（原型的「预计团款（元）」单字段将废弃）。

| # | 客户原话（摘要） | 问题本质 | PRD 修订 |
|---|----------------|----------|----------|
| 1 | 「预计团款多少元」不对；要有**单价**栏目，旁边再搞**备注** | 团款不是单一总价，而是**多项目组成**；用户自填类型（如酒店）+ 价格 + 备注 | 新增 **3.5.1 团款组成**、`project_quote_items` 表；去掉「预计团款」单输入框 |
| 2 | 澄清：是**一个人的单价**或**一组客人的价格**，**不是一个团的价格** | 计价维度是人均或每组，非整团 lump sum | 每行增加 `pricing_unit`：`per_person`（× 人数）\| `per_group`（× 组数，默认 1） |
| 3 | 应该还有**拼出**——收进来自己不发的，给出去；要加**同行旅行社** | 存在「收团 → 转交同行承接」业务线，与自营团并列 | 新增 `biz_type` + `outsourced_to_partner_id`；合作方增加 `partner_kind` 区分组团社/同行 |

**实现优先级**：团款组成与拼出均为 **P1**（客户明确表示建团环节即需使用），不推迟至 P2 计划毛利阶段。

#### 3.5.1 团款组成（分项单价 + 加减项，非整团总价）

**业务规则（客户反馈确立，v2.3 增补加减方向）**：

1. **禁止**在基本信息区只填一个「预计团款 / 合同总价」——行业实操是多项报价叠加。
2. 每一行由 OP **自填费用类型**（文本，如「团款」「门票」「优惠退费」「增加住宿」；可提供常用快捷项但不强制枚举）。
3. 每行必填 **加减方向** `direction`：
   - `add`（加项，默认）：在团款基础上增加收入，如团款、门票、增加住宿。
   - `subtract`（减项）：在团款基础上扣减，如优惠退费、退团赔款。
4. 每行必填 **单价（元）** 与 **计价方式**：
   - `per_person`（人均）：行金额 = 单价 × `(pax_adult + pax_child)`，人数变更时自动重算展示值。
   - `per_group`（每组客人）：行金额 = 单价 × `quantity`（组数，默认 1）；适用于「这一桌/这一车」打包价。
5. 每行选填 **备注**（如「标间含早」「早鸟优惠」），紧邻单价展示。
6. **计划收入合计**（只读）：

```
行金额 = unit_price × 数量因子（人均用人数，每组用 quantity）
计划收入 = Σ(direction=add 的行金额) − Σ(direction=subtract 的行金额)
```

作为后续计划毛利、定金尾款模板默认基数的来源（取代原 `budget_income_cents` 单字段）。

**UI 列顺序**：类型 | **+/−** | 单价 | 计价（人均/每组）| 数量 | 备注

**示例行**（32 人团）：

| 类型 | +/− | 单价 | 计价 | 行金额 |
|------|-----|------|------|--------|
| 团款 | + | ¥1,950/人 | 人均 | ¥62,400 |
| 门票 | + | ¥120/人 | 人均 | ¥3,840 |
| 优惠退费 | − | ¥50/人 | 人均 | −¥1,600 |
| 增加住宿 | + | ¥200/人 | 人均 | ¥6,400 |
| **计划收入** | | | | **¥71,040** |

**校验**：
- `subtract` 行单价仍须 > 0（减的是金额，不是负单价）。
- 合计为负时展示警告「计划收入为负，请核对加减项」，**不阻断保存**（地接社可能存在退团赔款场景）。

**UI 空状态**：无行时显示「添加团款明细，单价按人或按组填写，可加可减，不是整团一口价」+ `[+ 添加一行]`。

**与收支的关系**：团款组成是 **计划/报价层**；`transactions` 仍是 **实际发生层**。二者可不一致，P2 用偏差率对比（见 4.4）。

#### 3.5.2 拼出团（转交同行）

**业务规则**：

1. `biz_type=self_operated`（默认）：本社收操，与 v2.1 行为一致。
2. `biz_type=outsourced_out`（拼出）：本社从组团社收团，**转交同行地接社执行**；必须选择 `outsourced_to_partner_id`（`partner_kind` 为 `peer` 或 `both`）。
3. 拼出团列表展示「拼出」标签，并展示 **拼出待付**、**拼出结账状态**（见下表）；详情顶栏展示四维财务摘要（见 3.5.3）。
4. 拼出团使用 **3.5.3 拼出分项表** 记录应付同行明细；`outsourced_to_partner_id` 为团级承接同行，保存时分项行 `peer_partner_id` **统一继承**（v2.6 UI 不每行选同行）。
5. **拼出应付节点**（见 3.6）：保存拼出价格后可生成 `payment_schedules` 应付行，作为 **到期提醒 / 工作台逾期** 载体；**结账事实**以 `outsource_settlements` 为准（见 3.5.4）。
6. **拼出差价（预览）** = 团款组成计划收入（加−减）− 拼出预估合计（P1 仅展示）；实际结账以 settlements 累计为准。

**失败路径 3**：选择「拼出」但未选承接同行 → 阻止提交，提示「请选择拼出承接的同行旅行社」。

#### 3.5.3 拼出分项与顶栏四维摘要（仅拼出团）

**主路径（v2.6）**：建团/详情使用 **统一拼出分项表**——与团款组成上下分区（对组团社应收 vs 对同行应付）；顶栏实时展示 **四维财务摘要**：

```
[拼出] 天山同行地接 · 预估 ¥26,100 · 已付 ¥10,000 · 待付 ¥16,100 · 部分结清
```

**建团表单结构**（拼出时）：
1. 基本信息区 **不含** 承接同行（与人数/日期等通用字段并列）
2. **团款组成**（对组团社 · 计划收入）— 见 3.5.1
3. **拼出分项**（对同行 · 应付预估）— 区块内首行 **承接同行** 下拉 + 分项表，**禁止**基本信息内联单价/折叠「更多分项」

**计算公式**（顶栏、列表、数据层三处一致）：

```
预估总额 = Σ(project_outsource_items 各行金额)
已付金额 = Σ(outsource_settlements.amount_cents)
待付金额 = max(0, 预估总额 − 已付金额)
结账状态 = 已付=0 且 预估>0 → 未结清
         | 0<已付<预估 → 部分结清
         | 已付≥预估 → 已结清
```

- 无拼出明细时：预估为空，顶栏显示 `[拼出] 天山同行地接 · 待填拼出价`
- 明细变更导致预估变化时：待付重算；若已付 > 新预估，顶栏警告「已付超出预估，请核对」
- 结账状态为 **展示态**，不落库冗余枚举，由已付与预估实时推导

**列表筛选**：**拼出结账状态**（全部 / 未结清 / 部分结清 / 已结清），仅对 `biz_type=outsourced_out` 生效。

| 字段 | 说明 | 必填 |
|------|------|------|
| 承接同行 | 团级 `outsourced_to_partner_id`，分项表表头只读展示 | 是 |
| 费用项 | 自填文本，如「执行团款」「全陪段」；默认首行「执行团款」 | 是 |
| 单价 | 元/人 或 元/组 | 是 |
| 计价方式 | `per_person` / `per_group` | 是 |
| 数量 | 组数（人均时忽略，用团人数） | 条件 |
| 标准 | 执行标准说明 | 选填 |
| 备注 | 补充说明 | 选填 |

**拼出合计（预估）** = 各行金额之和（只读）；顶栏「预估」与此一致。

**拼出差价（预览）** = 团款组成计划收入（加项−减项）− 预估总额（P1 仅展示，不实算毛利）。

**保存后交互**：
- 弹窗提示「是否生成应付同行节点？」（与定金尾款模板并列）；允许稍后手动添加
- 拼出明细变更后，若已存在 `source_kind=outsource` 的 pending 节点，提示「是否同步金额」

**UI ASCII 示例**（建团 / 详情 Tab 内）：

```
拼出分项（对天山同行地接 · 应付预估）
| 费用项   | 单价  | 计价 | 标准           | 备注 | 小计    |
| 执行团款 | 1450  | /人  | 含车导餐不含票 |      | ¥26,100 |
| 拼出合计（预估） ¥26,100 · 拼出差价（预览）¥4,720              |
```

**交互**：
- 切换为「自营」时：拼出明细 Tab 折叠隐藏，已填数据保留不删（再次切回拼出可恢复）。
- 拼出但未填价格：允许保存，顶栏显示「待填拼出价」；收付款节点 Tab 提示可生成应付节点。

#### 3.5.4 拼出应付结账（v2.5）

**触发入口**：
- 详情顶栏「待付 > 0」时显示 **[应付结账]** 按钮
- 收付款节点 Tab 拼出行操作 **[结账]**（替代单纯「标记已付」，见 3.6）

**应付结账弹窗**：

```
应付结账
同行      天山同行地接（只读）
待结金额  ¥16,100.00（只读）
本次结算  ¥[16100.00]（默认=待结，可改，须 ≤ 待结）
结算方式  [银行转账 ▾]（bank_transfer|cash|alipay|other）
结算日期  [2026-06-18]
备注      [首笔拼出款]
                    [取消] [确定]
```

**业务规则**：
1. 本次结算金额须 > 0 且 ≤ 待付金额；超额阻止提交
2. 确认后写入 `outsource_settlements`，并 **联动创建** `transactions` 支出（`category=other`，`supplier_id` 为空，`note` 含同行名与结算方式）
3. 结账状态随已付累计自动推导，**不落库冗余枚举**
4. 当 `已付 ≥ 预估` 时，同步将 `source_kind=outsource` 的 pending `payment_schedules` 标记为 done（若存在）
5. 每次结账记录 `created_by`、`created_at`（v1 操作留痕，替代独立审计表）

**失败路径**：
- 待付为 0 仍点结账 → toast「该拼出已全部结清」
- 拼出无明细 → 阻止结账，引导先填价格

**b) 交互流程**

```

           --> 弹窗「还有 2 个未完成节点，确认全部视为已处理？」
               [逐个去处理] [确认结清]

           提示「合作方已停用，请重新选择或先启用」
```

## 第四章：超越竞品的差异化功能

### 4.1 收付款节点驱动的「结算感知」工作台

**竞品为何没有**：路书类工具的商业模式建立在「行程内容生产」上，财务只是行程的附属字段，其架构里没有「应收应付的时间维度」这一实体；传统 ERP 有应收应付，但其设计是给专职财务月底批处理用的，逾期信息埋在报表里，而中小地接社根本没有专职财务每天去翻报表。这是商业模式和目标组织规模差异导致的结构性盲区。

**本产品如何实现**：用独立的 `payment_schedules` 表（团 ID、方向、金额分、到期日、状态），工作台聚合查询 `status='pending'` 按 `due_date` 排序，逾期由日期比较实时推导而非定时任务写状态。选这个方案而不是 Cron Trigger 批量标记，因为推导式零运维、零延迟、且免费额度内不占 Cron 配额；代价是每次查询多一个日期比较，在千行量级可忽略。

**交互流程**：

```
计调建团时顺手添加节点「定金30% 到期06.10」「尾款 到期06.20」
    |
    v
06.11 老板打开工作台
    |
    v
首屏卡片「逾期应收 ¥23,500 / 3 笔」红色高亮
    |
    v
点击 --> 节点列表（按逾期天数降序）--> 点行进入团详情
    |
    v
打电话催款 --> 到账 --> 点「标记已收」--> 自动生成收入明细
    |
    v
工作台数字即时更新，逾期卡片消除红色
```

**风险与应对**：风险是用户嫌「建节点」多一步操作，节点数据稀疏导致功能空转。应对：新建团表单内提供「定金+尾款」一键模板（按比例自动算金额）；若上线后节点创建率仍低于 30%，备选方案是从收支明细反推（录入定金时询问「尾款何时收？」）。

### 4.2 团号为中心的统一信息模型

**竞品为何没有**：Excel 的本质是表格而非数据库，团、钱、合作方之间没有外键，只能靠人肉对齐行；路书的中心实体是「行程」，同一行程可复用于多个团，导致它的数据模型天然不适合做单团核算。两者都不是「没想到」，而是其基础数据模型决定了改造成本极高。

**本产品如何实现**：`group_no` 在租户内全局唯一（D1 复合唯一索引 `(org_id, group_no)`），收支、节点、合作方往来全部通过 `project_id` 外键挂团，顶栏搜索直达。团号自动生成 `JTT-YYYYMMDD-NN` 但允许手改，因为很多地接社已有自己的编号习惯，强制格式会成为迁移阻力。

**交互流程**：

```
微信里组团社问「JTT-20260601-02 尾款发你了」
    |
    v
计调在顶栏输入 0601 --> 下拉即时匹配（防抖 300ms）
    |
    v
回车进入团详情 --> 收付款 Tab --> 标记尾款已收
    |
    v
全程 < 10 秒，无需翻 Excel 文件名找「6月版最终版(2).xlsx」
```

**风险与应对**：风险是手改团号导致重复冲突的体验问题。应对：前端失焦即查重（而非提交时才报错）；若用户普遍手改，说明自动格式不符合行业习惯，备选方案是开放租户级前缀配置（已在 3.5 待决问题中预留）。

### 4.3 供应商成本溯源

**竞品为何没有**：ERP 的供应商模块面向采购合同管理，录入字段多达数十个，中小地接社用不动；Excel 里供应商只是支出行里的一个文本格子，同一家酒店被写成「悦榕」「西街悦榕」「悦榕酒店」三种名字，永远无法聚合。这是「重工具过重、轻工具无结构」的中间地带空缺。

**本产品如何实现**：供应商档案只有 4 个字段（名称、类型、联系人、电话），支出明细通过下拉**选择**而非手输供应商，从源头消灭别名问题。供应商详情页用一条 `SUM` 聚合出累计成本与未付金额。刻意不做采购合同、协议价库——那是 ERP 的领域。

**交互流程**：

```
记支出 --> 供应商下拉 --> 输入「悦」--> 命中「阳朔西街悦榕酒店」
    |                          |
    |                          +-- 无命中 --> 「+ 新建供应商"悦"」内联创建
    v
保存 --> 年底打开供应商详情
    |
    v
「本年度在悦榕消费 ¥128,400 / 23 个团」--> 拿着数据去谈明年协议价
```

**风险与应对**：风险是用户图省事不选供应商（该字段选填）。应对：选填是刻意的（强制会逼用户乱选污染数据）；若关联率低，在支出分类为「酒店/门票/用车」时将下拉前置为表单第二个字段，用顺手程度而非强制来提升关联率。

### 4.4 计划毛利 vs 实际毛利（P2 规划，行业方案核心财务能力）

**竞品为何没有**：Excel 里「报价」和「实际成本」往往分在不同 Sheet，或无报价只有决算；传统 ERP 的计划数在预算模块、实际数在核算模块，OP 看不到一屏对比。中小地接社需要的是 **团级别的轻量对比**，不是全公司年度预算系统。

**本产品如何实现（P2）**：计划收入由 `project_quote_items` 汇总（见 3.5.1），不再使用单一 `budget_income_cents` 字段。实际毛利仍由 `transactions` SUM 推导。团详情展示：

```
行金额 = unit_price × 数量因子（人均 × 人数，每组 × 组数）
计划收入 = Σ(add 行金额) − Σ(subtract 行金额)
计划毛利 = 计划收入 - 预算成本（可选 budget_cost_cents 或首版仅比收入）
实际毛利 = SUM(income) - SUM(expense)
偏差率 = (实际 - 计划) / 计划
```

拼出团（P2 预览）可增加 **计划差价** 一行：`计划收入 − project_outsource_items 预估总额`（见 3.5.3）。**对同行实际结账**以 `outsource_settlements` 累计已付为准（v2.5）；`payment_schedules`（`source_kind=outsource`）仅作到期提醒，done 状态在已付≥预估时同步。

偏差率 < -10% 时列表行标黄、工作台纳入「异常团」计数——对应行业方案「自动识别异常订单」。

**交互流程**：

```
OP 建团录入团款组成（团款 +1950/人、门票 +120/人、优惠退费 −50/人 …）合计 ¥71,040
    |
    v
行中陆续记账 --> 实际成本累加
    |
    v
团详情顶部：计划毛利 ¥12,000 | 实际毛利 ¥8,400 | 偏差 -30% ⚠
    |
    v
老板点开 --> 看支出明细找超支项（如用车追加）
```

**风险与应对**：风险是 OP 不填团款组成导致计划收入为空。应对：定金尾款模板默认取团款组成合计；仍为空则仅展示实际毛利（与 v2.1 一致）；建团页空状态文案强调「按单价录入，非整团一口价」。

**设计决策说明**：① 不设独立的「账户/钱包」实体——地接社关心的是「每个团赚不赚、谁欠谁」，不是银行账户余额，加账户体系会把产品推向记账软件红海。② `transactions.org_id` 冗余存储是刻意的：全局流水页直查一张表，避免 JOIN projects，D1 单表索引扫描性能可预期。③ 逾期不落库、纯推导，消灭「定时任务漏跑导致状态错」这一整类 bug。④ 刻意排除：审计日志表（v1 用 `created_by` 字段达到 80% 追溯效果）、软删除（`status=disabled` 已覆盖需求场景）。⑤ **团款组成与计划收入（v2.2/v2.3）**：`project_quote_items` 为计划层明细，含 `direction` 加减项；`projects` 不再存 `budget_income_cents`，计划收入 = Σ加 − Σ减，支撑 P2 计划 vs 实际毛利（见 4.4）。⑥ **拼出（v2.2/v2.3）**：`biz_type` + `outsourced_to_partner_id` 表达转交同行；`project_outsource_items` 独立记录拼出明细分项（给谁、单价、标准），避免与团款组成混用「对组团社报价」与「对同行成本」。⑦ **拼出应付节点（v2.4）**：`payment_schedules`（`source_kind=outsource`）作为到期提醒与工作台逾期载体。⑧ **拼出结账三层（v2.5）**：明细（`project_outsource_items`）→ 结账流水（`outsource_settlements`，已付事实）→ 节点（`payment_schedules`，提醒）；顶栏四维摘要由明细与流水实时聚合，结账状态不落库。

## 第六章：技术架构

```
+--------------------------------------------------------------------------+
| 浏览器层：React SPA                                                       |
| 职责：路由、表单校验(zod)、乐观更新、金额分<->元换算只在展示层发生           |
+--------------------------------------------------------------------------+
                          | HTTPS / JSON，Authorization: Bearer <JWT>
                          v
+--------------------------------------------------------------------------+
| 边缘层：Cloudflare Pages Functions（Hono 单 Worker）                      |
| 职责：JWT 验签、org_id 注入与强制过滤、zod 二次校验、限流、业务规则          |
| 约束：任何 SQL 必须带 org_id 条件——由 DAO 层统一拼接，handler 不写裸 SQL    |
+------------------------------+-------------------------------------------+
                |                              |
                v                              v
+------------------------------+  +---------------------------------------+
| D1 (SQLite)                  |  | KV                                    |
| 职责：唯一事实源              |  | 职责：refresh token(7d TTL)、          |
| 11 张表 + 复合索引            |  | 登录限流计数(15m)、dashboard 缓存(60s) |
| 迁移：drizzle-kit 生成 SQL    |  | 约束：KV 数据丢失不影响正确性，只影响    |
|                              |  | 体验（缓存重建 / 重新登录）              |
+------------------------------+  +---------------------------------------+
```

| 库名 | 用途 | 为何优于替代方案 | 大致包体积 |
|------|------|-----------------|-----------|
| hono | 边缘路由框架 | 为 Workers 设计，冷启动开销远小于 Express 类；中间件模型适合做 org_id 强制注入 | ~14KB |
| drizzle-orm | D1 ORM + 迁移 | 原生支持 D1 驱动；类型安全的查询构建器使「漏带 org_id」在编译期可被封装拦截；Prisma 不支持 D1 免费层友好部署 | ~30KB（按需） |
| zod | 前后端共享校验 | 同一 schema 两端复用，金额/日期规则只写一次；比 yup 的 TS 推导更完整 | ~14KB |
| react + react-router | SPA 框架 | 团队与 AI 生态最成熟；本产品页面数 ~12，不需要 SSR（数据全在登录墙后，无 SEO 诉求） | ~45KB + ~12KB |
| @tanstack/react-query | 服务端状态 | 内置缓存失效、乐观更新、请求去重，手写这些是 bug 高发区 | ~13KB |
| dayjs | 日期处理 | 到期日/逾期天数计算；比 date-fns 体积小且 API 对中文团队更直觉 | ~3KB |

不引入组件库（用 CSS 原生变量 + 少量手写组件），因为参考 UI（路书风格青绿主题）定制度高，antd 类库的覆盖样式成本反而更高。图表 v1 只有工作台一处趋势，用原生 SVG 手绘折线，不引入 echarts（~400KB）。

**最大架构风险**：D1 免费层的限制（单库 500MB、每日读写行数配额）与单区域写入延迟。应对：本产品数据天然轻量（一个 10 人地接社年数据量 < 5MB），按当前模型 500MB 可支撑数百租户；所有列表强制 cursor 分页（禁 OFFSET）+ 复合索引，把每请求读行数控制在两位数；若单库逼近上限，预案是按 org 分库（D1 支持多库绑定），数据模型已按 org_id 隔离，迁移只需搬运不需改 schema。

## 第九章：开发优先级

**P0 — 没有这个，产品根本无法使用。** 交付标准：功能可用，不需要完美。

- 注册（建企业）/ 登录 / JWT + refresh 续期
- 合作方 CRUD（建团的前置依赖）
- 接团单：新建、列表（基础筛选）、详情、状态流转
- **团款组成** CRUD（分项单价 + 备注，人均/每组计价）
- **拼出团**（业务类型 + 承接同行选择）
- 收支明细 CRUD + 团内三合计（应收/成本/毛利）

**P1 — 没有这个，用户第一次体验后不会回来。** 交付标准：功能完整，体验流畅。

- 工作台（4 统计卡 + 待办节点 + 最近团）
- 收付款节点 + 逾期推导 + 工作台联动 + 标记联动生成明细
- 供应商档案 + 支出关联
- 邀请码加入企业 + 员工列表/停用
- 全局搜索（团号/合作方）+ 列表 cursor 分页
- **团款组成加减项** `direction`（add/subtract，计划收入 = Σ加 − Σ减）
- **拼出明细** `project_outsource_items` CRUD（给谁、单价、标准）
- **拼出顶栏内联价** + **拼出应付节点**生成与金额同步（`source_kind=outsource`）
- **拼出应付结账**：顶栏四维摘要 + 结账弹窗 + `outsource_settlements` CRUD 与支出联动（见 3.5.3–3.5.4）

**P2 — 有了这个，用户会把产品推荐给别人。** 交付标准：稳定且有辨识度。

- 合作方/供应商详情页的往来聚合（历史团 + 累计金额）
- XLSX/CSV 导出（8.1 全部三种）
- 定金+尾款一键节点模板；收支「复制为新支出」
- 工作台毛利趋势图（近 6 个月）
- **计划 vs 实际毛利**（`project_quote_items` 加减汇总 + 偏差展示，见 4.4）
- **拼出团计划差价预览**（计划收入 − 拼出预估，见 3.5.3）
- **拼出列表结账状态筛选**、**工作台拼出待结汇总**（见 3.2、3.5.3；列表筛选原型已实现，工作台汇总 P2）
- **应收应付账龄简表**（按合作方/供应商聚合节点逾期天数）

**P3 — 有了这个，一部分用户会付费或强烈倡导。** 交付标准：精致且有完整文档。

- 批量对账单导出（zip）
- 团号前缀自定义、member 细粒度权限
- 移动端布局完整适配（≤ 768px）
- 数据导入（Excel 历史台账迁移向导）
- **询价/报价单 PDF 导出**（行业方案「一键报价」的轻量版）
- **导游端小程序 MVP**（只读行程+名单+移动报账入口，见 1.5）
- **老板 BI 增强**（现金流趋势、异常团列表、供应商 TOP 成本）

## 第十一章：开发者交接说明

本章写给将要实现本产品的你（AI 编程助手）。

### a) 实现顺序建议

按此顺序构建：**① schema + 迁移 → ② 认证与 org 中间件 → ③ 合作方（含 partner_kind）→ ④ 接团单 + 团款组成（含加减项）+ 拼出明细 + 顶栏内联价 → ⑤ 拼出应付节点联动 → ⑥ 拼出结账流水（`outsource_settlements` + 3.5.4 弹窗）→ ⑦ 收支 → ⑧ 节点 → ⑨ 工作台 → ⑩ 供应商 → ⑪ 员工/邀请 → ⑫ 导出**。

### b) 最可能导致返工的三个决策

1. **金额换算的位置**。安全默认：API 进出全部用 `_cents` 整数，元<->分换算只发生在前端展示组件这一层。改向信号：如果你发现自己在后端写 `* 100` 或 `/ 100`，说明换算层泄漏了，立即收拢。
2. **org_id 过滤的实现方式**。安全默认：写一个 DAO 层（或 drizzle 查询包装器），所有查询必须经它注入 `org_id`，handler 禁止裸写 SQL/查询构建器。改向信号：任何一个接口测试中出现跨租户数据，停下重构这一层，而不是在 handler 里补 where。
3. **节点与明细的联动**（3.6 待决问题的已定方案）。安全默认：标记节点 done 时在同一 D1 batch 内创建明细并回填 `done_txn_id`。改向信号：如果撤销/删除的级联逻辑让你写出超过 ~50 行的状态机，简化为「联动创建，撤销时只提示不自动删」。

### c) 哪里要严格，哪里可以灵活

| 章节 | 性质 |
|------|------|
| 1.6 范围边界 | **约束**——不得实现「明确排除」表中的项；「后续版本规划」表中的项不得提前做到 P0/P1 |
| 第三章各模块的状态清单与失败路径 | **约束**——状态枚举与流转规则不得增删 |
| 第三章 ASCII 图的具体像素/布局 | 建议——以原型 `prototype/index.html` 风格为准，可优化 |

### d) 已知的未知项
